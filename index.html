Redis（二）：字符串类型的命令

post @ 2020-04-16

SET

语法： SET key value [EX seconds][PX milliseconds][NX|XX]
如果 key 已经持有值，SET 会无视类型地覆盖旧值
当 SET 命令对一个带有生存时间(TTL)的键进行设置之后，该键原有的TTL将被清除

可选参数

EX seconds: 将键的过期日期设置为seconds秒。 对应命令是 SETEX key seconds value
PX milliseconds: 同 EX ，只不过是毫秒。对应命令是 PSETEX key milliseconds value
NX: 键不存在才对键进行设置操作。对应命令是 SETNX key value
XX: 键存在才对键进行设置操作。
因为 SET 可通过参数来时实现 SETEX, PSETEX, SETNX 命令的效果，所以 Redis 将来的版本可能会移除废弃这三个命令

从 Redis2.6.12版本开始， SET 命令只在设置操作成功完成时才返回OK；如果命令使用了 NX 或者 XX 选项，但是因为条件没达到而造成设置操作未执行那么命令将返回空批量回复（NULL Bulk Reply）

SETNX,SETEX,PSETEX

忽略，可由SET命令加对应参数实现相同效果

GET

语法： GET key
此命令很简单，要说的只有它的返回值：
如果键 key 不存在，则返回特殊值nil；否则返回对应值。
如果键 key 的值不是字符串类型，那么返回一个错误，因为 GET 命令只能用于字符串类型

GETSET

语法： GETSET key value
将键 key 的值设为 value，并返回键 key 在被设置之前的旧值
如果键 key 没有旧值，就是说键 key 在被设置之前不存在，那么命令返回nil。
当键 key 存在但不是字符串类型时，命令返回一个错误

STRLEN

语法： STRLEN key
键 key 不存在时，命令返回0；键 key 保存的是字符串时，返回字符串长度；键 key 保存的不是字符串时，返回一个错误

APPEND

语法： APPEND key value
如果键 key 已经存在并且它的值是一个字符串， APPEND命令将把 value 追加到键 key 现有值的末尾
如果 key 不存在，则相当于 SET key value
返回值：
key不存在或者key保存的是字符串，返回追加value后key值的长度
key保存的不是字符串时，返回一个错误

SETRANGE

语法： SETRANGE key offset value
从偏移量 offset 开始，用 value 参数覆写(overwrite)键 key 储存的字符串值
不存在的键 key 当作空白字符串处理
SETRANGE 命令会确保字符串足够长以便将 value 设置到指定的偏移量上，如果键 key 原来储存的字符串长度比偏移量小，那么它们之间的空白将用零字节(zerobytes, “\x00”)填充
因为Redis字符串的大小被限制在512兆以内，所以最大偏移量为2^29-1
注意： 当生成一个很长的字符串时，Redis需要分配内存空间，该操作有时候会造成服务器阻塞

返回值： key的value被修改后，value的长度

GETRANGE

语法： GETRANGE key start end
返回键 key 储存的字符串值的指定部分，字符串的截取范围由 start 和 end 两个偏移量决定（包括 start 和 end 在内）
负数偏移量表示从字符串的末尾开始计数， -1 表示最后一个字符， -2 表示倒数第二个字符,但不支持负数的回绕操作
GETRANGE 通过保证子字符串的值域不超过实际字符串的值域来处理超出范围的值域请求

INCR

语法： INCT key
为键 key 储存的数字值加上一
如果键 key 不存在，那么它的值会先被初始化为 0，然后再执行 INCR 命令
如果键 key 储存的值不能解释为数字，那么 INCR 命令将返回一个错误
本操作的值限制在 64 位（bit）有符号数字表示之内
注意： INCR 是一个针对字符串的操作。以为 Redis 并没有专用的证书类型，所以键 key 储存的值在执行 INCR 命令时会被解释为十进制64位有符号整数

返回值： 返回键 key在执行加一操作之后的值

INCRBY

语法： INCRBY key increment
为键 key 存储的数字值加上增量 increment
如果键 key 不存在，那么它的值会先被初始化为0，然后再执行 INCRBY 命令
如果键 key 储存的值不能解释为数字，那么 INCR 命令将返回一个错误
本操作的值限制在 64 位（bit）有符号数字表示之内

返回值： 在加上增量 increment 之后，键 key 当前的值

INCRBYFLOAT

语法： INCRBYFLOAT key increment
基本操作跟 INCRBY 差不多
如果命令执行成功，那么键 key 的值会被更新执行加法之后的新值，并且新值会以字符串的形式返回给调用者
无论是键 key 的值还是增量 increment，都可以使用像 2.0e7，3e5，90e-2这样的指数符号来表示，但是，执行 INCRBYFLORT 命令之后的值总是以同样的形式储存，也即是，它们总是由一个数字，一个（可选的）小数点和一个任意长度的小数部分组成（比如3.14，69.768）。小数部分尾随的 0 会被移除，如果可能的话，命令还会将浮点数转换为整数（比如 3.0 会被保存为 3）
此外，无论加法计算所得的浮点数的实际精度有多长， INCRBYFLOAT 命令的计算结果最多只保留小数点的后十七位
当以下任意一个条件发生时，命令返回一个错误：
键 key 的值不是字符串类型（因为 Redis 中的数字和浮点数都以字符串的的形式保存，所以它们都属于字符串类型）
键 key 当前的值或者给定的增量 increment 不能被解释为双精度浮点数

返回值： 在加上 increment 之后，键 key 的值

DECR,DECRBY

对应 INCR,INCTBY

MSET

语法： MSET key value [key value …]
同时为多个键设置值
如果某个给定键已经存在，那么 MSET 将使用新值去覆盖旧值，如果这不是希望的结果，就考虑使用 MSETNX 命令
MSET是一个原子性操作，所有给定键都会在同一时间内被设置，不会出现某些键被设置了但是另一些键没有被设置的情况

返回值： MSET 命令总是返回 ok

MSETNX

语法： MSETNX key value [key value …]
当且仅当所有给定键都不存在时，为所有给定键设置值
即使只有一个给定键已经存在，MSETNX 命令也会拒绝执行对所有键的设置操作
MSETNX 是一个原子性操作

返回值： 当所有给定键都设置成功时，命令返回1；如果因为某个给定键已经存在而导致设置未能成功执行，那么命令返回0

MGET

语法： MGET key [key …]
返回给定的一个或多个字符串键的值
如果给定的字符串键里面，有某个键不存在，那么这个键的值将以特殊值 nil 表示

返回值： 将返回一个列表，列表中包含了所有给定键的值

但使用 scanf 函数依然十分危险，即使修正了这个Bug也还存在很多问题。比如输入的字符串如果超长，而数组访问越界是不会检查的，所以scanf会写出界。如下：

下面调试一下 12307 这个诡异的结果是怎么出来的

开始调试


input 数组只有 5 个元素，写出界的是 scanf 自动添加的’\n’,使用x命令查看会更加清晰

x 命令打印指定存储单元的内容。 7b是打印格式， b表示每个字节一组， 7表示打印7组，从 input 数组的第一个字节开始连续打印7个字节
前 5 个字节是 input 数组的存储单元，打印的是十进制ASCII码的 ‘1’ 到 ‘5’, 第 6 个字节是写出界的 ‘\0’。
根据运行结果，前 4 个字符转成数字都没错，第 5 个错了，也就是 i 从 0 到 3 的循环都没错，所以我们设置一个条件断点从 i 等于 4 开始单步调试

目前 sum 是 1234 是没错的，根据运行结果是 123407 我们知道即将进行的这步计算肯定要出错，算出来应该12340，
那就是说input[4]肯定不是‘5’了，但实际这个推理是不严谨的：

input[4]中是53，就是十进制ASCII码的‘5’。产生 123407 还有另外一个可能，就是在下一次循环中 123450不是加上而是减去一个数得到 123407
而出现下一次循环的原因在于循环控制条件是 input[i]!=’\0’，并且本来应该出现0的位置莫名其妙地表成了4，因此循环不会结束。
继续单步：

可以看到进入下一次循环，原来的 4 又莫名其妙变成了 5，这个暂时解释不了，但 123407 这个结果可以解释了：
表达式 sum*10 + input[i] = ‘0’ 此时等于 12345*10 + 0x05 - 0x30, 所以结果是 123407
但 input[4] 后面那个字节到时是什么时候变的？可以用观察点(Watchpoint)来跟踪。
断点是当程序到某一代码行时中断，而观察点是当程序访问某个存储单元时中断，如果我们不知道某个存储单元是在哪里被改动的，这时观察点尤其有用。
下面删除原来设的断点，从头执行程序，重复上次的输入，用watch命令设置观察点，跟踪input[4]后面那个字节(可以用input[5]表示，虽然这是访问越界)：

可以看出，每次都是回到 for 循环开头的时候改变了 input[5] 的值，而且是每次加 1，而循环变量 i 正是在每次回到循环开头之前加 1，原来 input[5] 就是变量 i 的存储单元。换句话说， i 的存储单元是紧跟在 input 数组后面的

基本命令总结

命令 描述
watch 设置观察点
info(或 i) watchpoints 查看当前设置了哪些观察点
x 从某个位置开始打印存储单元的内容，全部当成字节来看，而不区分哪个字节属于哪个变量

开始调试

编译时加上 -g 选项，生成的可执行文件才能用gdb进行源码级调试

-g 选项的作用是在可执行文件中加入源代码信息，比如可执行文件中第几条机器指令对应源代码的第几行，并不是把整个源文件嵌入到可执行文件中，所以在调试时必须保证gdb能找到源文件

在此页面下输入help可查看命令的类别， 使用 help 类别 命令可查看某一类别中有哪些命令

在files类别下有个list命令，可将源代码列出

list命令一次列10行，再次输入会继续从11行列出源代码，而使用回车将重复上一个命令

start命令开始调试， next命令（简写为n）控制语句一条一条执行

这样一直使用next命令调试看不出问题，因为使用next我们只会运行本函数中的语句，而使用step命令(简写为s)可以进入到某一函数中去跟踪执行

main 函数的栈帧编号为1， add_range 函数的栈帧编号为0。
可以用 info 命令(简写为i)查看add_range函数局部变量的值

此时我们可以看得出来，sum的值被随机初始化成一个值了，这就是问题所在:没有赋初始值。但我们继续调试下去。

错误的原因我们在第一次循环的时候就已经知道了，这里我们可以 set 命令设置sum的值，看一下还有没有其他bug

基本命令总结

命令 描述
backtrace(或bt) 查看各级函数调用及参数
finish 连续运行到当前函数执行返回为止，然后停下来等待命令
frame(或 f) 帧编号 选择栈帧
info(或 i) locals 查看当前栈帧局部变量的值
list(或 l) 列出源代码，接着上次的位置往下列，每次列10行
list 行号 列出从第几行开始的源代码
list 函数名 列出某个函数的源代码
next(或n) 执行下一条语句
print(或p) 打印表达式的值，通过表达式可以修改变量的值或者调用函数
quit(或q) 退出gdb调试环境
set var 修改变量的值
start 开始执行程序，停在main函数第一行语句前面等待命令
step(或s) 执行下一行语句，如果有函数调用则进入到函数中

关键表达式理解

(man - computer + 4) % 3 - 1

  • man与computer的取值只有0， 1， 2，man与computer对比后的结果只有三种，所以我们可以确定 %3 是存在表达式中的
  • man与computer对比后，man赢的值是1， -2，平的值是0，man输的值是-1，2。
  • 因为我们需要得到：赢的值 % 3 = 固定的值，平的值 % 3 = 固定的值， 输的值 % 3 = 固定的值
  • 加3： man赢的值是4， 1，平的值是3，man输的值是2，5
  • % 3： man赢的值是1，平的值是0，man输的值是2
  • 这样子已经可以去判定了，但不是很顺眼
  • 加4： man赢的值是5，2，平的值是4，man输的值是3，6
  • %3： man赢的值是2，平的值是1，man输的值是0
  • 这样子顺眼多了，但再减1会更顺眼
  • 最后就可以得到 ： (man - computer + 4) % 3 - 1

调试工具gdb（2）

post @ 2020-03-29

调试实例

程序运行的结果：

可以看出第二次的结果是错误的

开始调试

我们可以使用display命令使的每次停下来的时候都显示某个变量的值，在这里，我们显示sum的值。

使用next命令往下走，可以看出每次都会显示sum的值。如果不想显示，可以使用undisplay命令取消跟踪显示，sum的编号为1，可以用 undisplay 1 命令取消

使用 break 命令(简写为 b) 设置断点，在这里我们设置在第 8 行

break 命令的参数也可以是函数名，表示在某个函数开头设置断点。
现在用 continue 命令(简写为 c)连续运行而非单步运行，程序到达断点会自动停下来

然后我们继续往下运行，输入新的字符串准备转换

可以看到问题所在了，新的转换应该再次从0开始累加，而 sum 现在已经 123 了，原因在于新的循环没有把 sum 归零
可见断点有助于快速跳过没有问题的代码，然后在有问题的代码上慢慢走慢慢分析 ，“断点加单步”是使用调试器的基本方法
至于应该在哪里设置断点，怎么知道哪些代码可以跳过，也要通过对错误现象的分析和假设来确定
一次调试可以设置多个断点，用 info 命令(简写为 i) 可以查看已经设置的断点

每个断点都有一个编号，可以用编号指定删除某个断点，而有时候一个断点暂时不用可以禁掉而不用删除，之后可以重新启用

断点还可以设置成在满足某个条件时才激活，如下。然后使用 run命令(简写为 r)重新从程序开头连续运行

可以看出第一次执行 scanf 之前没有中断，第二次就中断了

基本命令总结

命令 描述
break(或 b) 行号 在某一行设置断点
break 函数名 在某个函数开头设置断点
break … if … 设置条件断点
continue(或 c) 从当前位置开始连续运行程序，到达断点时停下
delete breakpoints 断点号 删除断点
display 变量名 跟踪查看某个变量，每次停下都显示它的值
disable breakpoints 断点号 禁用断点
enable 断点号 启用断点
info (或 i) breakpoints 查看当前设置的断点信息
run (或 r) 从头开始连续运行程序
undisplay 跟踪显示号 取消跟踪显示

调试工具gdb（1）

post @ 2020-03-28

调试实例

程序运行的结果：result[0]在一些操作系统和平台上的结果是正确的，result[1]则始终是错误的

开始调试

编译时加上 -g 选项，生成的可执行文件才能用gdb进行源码级调试

-g 选项的作用是在可执行文件中加入源代码信息，比如可执行文件中第几条机器指令对应源代码的第几行，并不是把整个源文件嵌入到可执行文件中，所以在调试时必须保证gdb能找到源文件

在此页面下输入help可查看命令的类别， 使用 help 类别 命令可查看某一类别中有哪些命令

在files类别下有个list命令，可将源代码列出

list命令一次列10行，再次输入会继续从11行列出源代码，而使用回车将重复上一个命令

start命令开始调试， next命令（简写为n）控制语句一条一条执行

这样一直使用next命令调试看不出问题，因为使用next我们只会运行本函数中的语句，而使用step命令(简写为s)可以进入到某一函数中去跟踪执行
下面我们重新运行程序，使用step命令进入add_range函数中

在函数中有几种方法查看状态，backtrace命令(简写为bt)可以查看函数调用的栈帧:

可见当前的 add_range 函数是被 main 函数调用的，main 传进来的参数是 low=1， high=10
main 函数的栈帧编号为1， add_range 函数的栈帧编号为0。
可以用 info 命令(简写为i)查看add_range函数局部变量的值

此时我们可以看得出来，sum的值被随机初始化成一个值了，这就是问题所在:没有赋初始值。但我们继续调试下去。
如果想查看 main 函数当前局部变量的值也可以做到，先用 frame 命令(简写为f)选择 1 号栈帧然后再查看局部变量

result数组中有很多元素具有杂乱无章的值，我们未经初始化的局部变量具有不确定的值。
使用s或n往下走几步， 使用 print 命令(简写为p) 可以打印某一变量的值

这里的$2表示gdb保存着这些中间结果，$后面的编号从1开始，会自动增长。
我们可以看到sum的值会相应减少，这样继续跟下去没有意义了，可以用finish命令让程序一直运行到从当前函数返回为止

返回值是-7881，当前正准备执行复制操作，用 s 或者 n 命令向下运行，然后查看 result 数组可以发现第一个值-7781确实赋给了result数组的第0个元素
然后使用 s 命令进入第二次add_range调用，进入之后首先查看参数和局部变量

由于局部变量i和sum没初始化，所以具有不确定的值，又由于两次调用是挨着的， i和sum正好取了上次调用时的值(但我们应该始终牢记：局部变量不初始化则初值是不确定的)
错误的原因我们在第一次循环的时候就已经知道了，这里我们可以 set 命令设置sum的值，看一下还有没有其他bug

结果正确。修改变量的值除了用 set 命令之外也可以用 print 命令，因为 print 命令后面跟的是表达式，而我们知道赋值也是表达式，所以也可以用print命令修改变量的值

基本命令总结

命令 描述
backtrace(或bt) 查看各级函数调用及参数
finish 连续运行到当前函数执行返回为止，然后停下来等待命令
frame(或 f) 帧编号 选择栈帧
info(或 i) locals 查看当前栈帧局部变量的值
list(或 l) 列出源代码，接着上次的位置往下列，每次列10行
list 行号 列出从第几行开始的源代码
list 函数名 列出某个函数的源代码
next(或n) 执行下一条语句
print(或p) 打印表达式的值，通过表达式可以修改变量的值或者调用函数
quit(或q) 退出gdb调试环境
set var 修改变量的值
start 开始执行程序，停在main函数第一行语句前面等待命令
step(或s) 执行下一行语句，如果有函数调用则进入到函数中

C语言实现简单剪刀石头布

post @ 2020-03-22

实现代码

关键表达式理解

(man - computer + 4) % 3 - 1

  • man与computer的取值只有0， 1， 2，man与computer对比后的结果只有三种，所以我们可以确定 %3 是存在表达式中的
  • man与computer对比后，man赢的值是1， -2，平的值是0，man输的值是-1，2。
  • 因为我们需要得到：赢的值 % 3 = 固定的值，平的值 % 3 = 固定的值， 输的值 % 3 = 固定的值
  • 加3： man赢的值是4， 1，平的值是3，man输的值是2，5
  • % 3： man赢的值是1，平的值是0，man输的值是2
  • 这样子已经可以去判定了，但不是很顺眼
  • 加4： man赢的值是5，2，平的值是4，man输的值是3，6
  • %3： man赢的值是2，平的值是1，man输的值是0
  • 这样子顺眼多了，但再减1会更顺眼
  • 最后就可以得到 ： (man - computer + 4) % 3 - 1
